# Births, LLM Eulogies & Playtime-Gated Bans — Design

Date: 2026-06-14
Status: Approved (pending spec review)

## Goals

Three related features, unified by a single "real life" lifecycle model:

1. **Playtime-gated banning** — only ban a player when they die after **≥ 60 minutes of playtime**
   (Σ session time, not wall-clock) on that life. Sub-60-minute deaths still end the life and start
   a new one on respawn, but issue no ban.
2. **Birth announcements** — when a new life "sticks", post a substantial, funny, newspaper-style
   birth notice to a dedicated **births channel**.
3. **LLM eulogies** — when a real life ends in death, post a substantial, funny-and-roasty,
   newspaper-style **obituary** to a dedicated **eulogy/deaths channel**, generated by an OpenRouter
   LLM from a rich blend of raw ADM log + structured facts. **This replaces the existing death
   feed.**

De-duplication of spawn-reroll suicides (players who kill themselves repeatedly to get a better
spawn) is a first-class requirement, handled by the grace threshold below.

## The "real life" lifecycle model (core idea)

A life passes through gates as its `playtime_seconds` accrues:

| Playtime in a life      | Birth announced?   | Eulogy on death? | Ban on death? |
| ----------------------- | ------------------ | ---------------- | ------------- |
| < 5 min (reroll-suicide)| no                 | no               | no            |
| 5 min – 60 min          | **yes** (~5 min in)| **yes**          | no            |
| ≥ 60 min                | yes                | yes              | **yes**       |

- A life **"counts"** once `playtime_seconds ≥ LIFE_GRACE_MINUTES` (default **5 min**). This single
  threshold gates both births and eulogies and is what kills spawn-reroll noise: rapid suicides
  never reach 5 minutes of playtime, so they are never announced and never eulogized.
- **Banning** uses a separate, higher gate: `playtime_seconds ≥ BAN_MIN_PLAYTIME_MINUTES`
  (default **60 min**). This is feature 1.
- Both gates are additionally subject to:
  - **`go_live_at` gating** — never retro-announce or retro-ban backfilled history (reuses the
    existing `bot_state.go_live_at` pattern that `DeathBanService` already uses).
  - **Freshness gating** — `LIFECYCLE_MAX_AGE_MINUTES` (default **30 min**) suppresses a
    post-downtime backlog from flooding the channels (mirrors `DEATH_FEED_MAX_AGE_MINUTES`).

### Why playtime, not wall-clock

`LifeTracker::death()` closes the open `game_session` **before** ending the life
(`app/Services/Life/LifeTracker.php:41-43`), so `lives.playtime_seconds` is fully accurate at the
instant of death. Both the 5-minute grace and the 60-minute ban gate read it directly. For an
**open** life still being judged for a birth, elapsed playtime is computed with the existing
`LivePlaytime` helper (`app/Services/Life/`).

### Worked example (the reroll case)

Player suicides 3× in 2 minutes, then settles:

```
spawn → life A → suicide @ 40s   (playtime 40s  → no birth, no eulogy, no ban)
spawn → life B → suicide @ 25s   (playtime 25s  → no birth, no eulogy, no ban)
spawn → life C → ...survives...  (playtime ≥5m  → birth announced ~5m in)
                 ...dies @ 47m    (playtime 47m  → eulogy; no ban, under 60m)
```

Net: **one** birth (for C), **one** eulogy (C's death), **zero** bans. Exactly the desired de-dup.

## Data model

Migration adding three columns to `lives`:

| column              | type                  | notes                                                    |
| ------------------- | --------------------- | -------------------------------------------------------- |
| death_log           | text, nullable        | captured raw ADM excerpt around the death (LLM color)    |
| birth_announced_at  | timestamp, nullable   | set when the birth notice is posted; null = not yet      |
| eulogy_posted       | boolean, default false| idempotency marker for the obituary                      |

No new tables. `birth_announced_at` doubles as the birth idempotency marker (non-null = announced).

## Capturing the raw log (hybrid LLM context)

The LLM gets **both** raw log color and reliable structured facts.

**Raw capture (`AdmIngestor`):** while processing a file's lines in chronological order, the
ingestor maintains a small **in-memory per-player ring buffer** (last ~50 lines / ~15 min). When a
death line closes a life, it snapshots that player's recent raw lines + the kill/death line into
`lives.death_log`.

- No per-line DB writes (the buffer is in-memory); only one write, on death.
- Deaths and their preceding hits/positions sit in the **same ADM file**, so the death window is
  captured cleanly. The rare file-boundary case (a death in the first lines of a new file) loses a
  little preceding color but still has the kill/death line itself (killer, weapon, distance) — an
  acceptable, bounded loss (YAGNI on cross-file stitching).

**Births reuse structured facts + the _previous_ life's `death_log`** (e.g. "last time a bear ate
you — welcome back"), so births need no separate raw buffer.

**Structured facts (`LifeFactsBuilder`, pure):** built from the `Life` model:

- wall-clock age (`ended_at − started_at`) **and** playtime age (`playtime_seconds`)
- death cause, killer gamertag, weapon, distance
- **associates left behind** — best-effort from the bounty associate detector
  (`app/Services/Bounty/AssociateDetector`); omitted if empty/unavailable
- linked-or-not (drives the mention), referrer, and a one-line summary of the player's prior death
  (for birth "comeback" framing)

## Output format & voice (newspaper-grade, not one-liners)

Each post is a **rich Discord embed styled like a newspaper article** — substantial, not a single
line — accompanied by a short plain-content line that carries the real mention ping.

- **Headline** — embed title; a punchy tabloid headline
  (e.g. *"LOCAL MAN, 47 MINUTES OLD, MEETS BEAR — BEAR UNAVAILABLE FOR COMMENT"*).
- **Body** — embed description; a multi-paragraph obituary / birth notice, **~150–350 words**
  (well within the 4096-char embed limit). Dateline, the story of how they lived/died/arrived, a
  roast, a fictional-witness quote, etc. Full Discord markdown (**bold**, *italics*, `> blockquote`,
  bullet lists) and liberal emojis (🕯️💀🐻⚰️🎉👶).
- **Vital statistics flourish** — embed fields or a styled footer carrying age (wall-clock +
  playtime), cause, weapon, distance, associates left behind.
- **Color & footer** — black/grey border for eulogies, green for births; footer like
  *"The One Life Tribune · died 12 minutes ago"*.
- **Gender/"it's a boy!" gimmick is dropped** — no character/skin/gender data exists in the ADM
  logs (confirmed: `AdmParser` regexes capture only gamertag, id, position, and death details).
  Birth notices lean on the comeback/respawn angle instead.

### Mentions (real pings + newspaper look)

Discord does **not** fire a notification for a mention placed *inside* an embed. Therefore:

- The real `<@id>` ping for a **linked** player goes in the **plain message content line above the
  embed** (e.g. `🕯️ <@id> has fallen.`). Unlinked players get no content ping.
- The LLM is instructed to reference the subject/killer with literal placeholders `{{PLAYER}}` /
  `{{KILLER}}`; we substitute **after** generation — a Discord mention for linked players, a
  backticked gamertag otherwise. This prevents the model from mangling `<@id>` syntax and follows
  the repo's "public channel posts mention linked players" convention
  (`app/Services/Lookup/PlayerMention`). If the model omits a required placeholder, we prepend the
  subject reference defensively.

## New subsystems

```
app/Services/Llm/
  OpenRouterClient.php          thin Http-facade wrapper (fakeable). Reads OPENROUTER_API_KEY,
                                OPENROUTER_MODEL (default anthropic/claude-haiku-4.5),
                                OPENROUTER_BASE_URL, OPENROUTER_TIMEOUT_SECONDS. Returns generated
                                text or throws.

app/Services/Lifecycle/
  LifeFactsBuilder.php          PURE: Life -> structured-facts array (ages, cause, killer, weapon,
                                distance, associates, linked, prior-death summary).
  AnnouncementGenerator.php     Builds the birth/eulogy prompt (facts + raw death_log), calls the
                                client, returns {headline, body}. On ANY failure (no API key,
                                timeout, HTTP error, empty/garbage output) -> canned fallback. The
                                "savage post-apocalyptic obituary columnist" persona + the
                                substantial/formatted/emoji-rich brief live in the system prompt.
                                max_tokens ~900; randomness injected for deterministic tests.
  LifecycleNotifier.php         interface { announceBirth(Life, payload); announceEulogy(Life, payload) }
  DiscordLifecycleNotifier.php  posts content-line + embed; births -> BIRTHS_CHANNEL_ID,
                                eulogies -> EULOGY_CHANNEL_ID. Best-effort (swallows failures).
  NullLifecycleNotifier.php     no-op (tests / unconfigured channels).
```

**Output contract:** the LLM returns a headline + body; convention is **first line = headline, rest
= body**. The canned fallback supplies both (eulogies reuse the existing `config/personality.php`
`death.*` pools; births get a new `birth.*` pool — both short, since fallback only appears during an
OpenRouter outage).

### Periodic driver

`app/Services/LifecycleAnnounceService.php` (extends `Laracord\Services\Service`, **60s**), runs
after ingestion each tick:

- **Due births:** lives where elapsed playtime ≥ grace, `birth_announced_at IS NULL`, started after
  `go_live_at`, and fresh (within `LIFECYCLE_MAX_AGE_MINUTES`) → generate → post →
  set `birth_announced_at`.
- **Due eulogies:** lives where `ended_at IS NOT NULL`, the life counted (`playtime_seconds ≥
  grace`), `eulogy_posted = false`, ended after `go_live_at`, and fresh → generate → post →
  set `eulogy_posted = true`.
- Gated by `LIFECYCLE_ENABLED`. Disabled / no API key still posts canned fallbacks (never silent).
- Edge: a life that reached grace but died before the birth fired posts birth **then** eulogy in the
  same tick ("born… and immediately laid to rest") — acceptable and on-theme.
- **Not gated by `BAN_DRY_RUN`** (channel posts are independent of real Nitrado bans, exactly like
  the death feed it replaces).

### Changes to existing code

- **`DeathBanService`** (`app/Services/Ban/`): add one clause —
  `where('playtime_seconds', '>=', BAN_MIN_PLAYTIME)` — and **remove** its `feed->died()` call.
  It keeps issuing the ban and the existing `ban.dm.death` DM. Eulogies now belong to
  `LifecycleAnnounceService`.
- **Retire `app/Services/DeathFeed/`** — eulogies replace it. The canned copy in
  `config/personality.php` `death.*` survives as the eulogy fallback. The `DiscordDeathFeedNotifier`
  wiring in `IngestAdmService` is removed.
- **`AdmIngestor`** — add the in-memory ring buffer + `death_log` snapshot on death (thread the
  captured excerpt into `LifeTracker::death()` or write it alongside).

## Config & env (new)

`config/lifecycle.php` and `config/llm.php`, defaults pinned in `phpunit.xml`:

| key                         | default                      | meaning                                  |
| --------------------------- | ---------------------------- | ---------------------------------------- |
| `BAN_MIN_PLAYTIME_MINUTES`  | `60`                         | ban only if playtime ≥ this              |
| `LIFE_GRACE_MINUTES`        | `5`                          | a life "counts" (birth/eulogy) at this   |
| `LIFECYCLE_ENABLED`         | `true`                       | master switch for births + eulogies      |
| `LIFECYCLE_MAX_AGE_MINUTES` | `30`                         | freshness gate for both                  |
| `BIRTHS_CHANNEL_ID`         | (unset → notifier no-ops)    | births channel                           |
| `EULOGY_CHANNEL_ID`         | (unset → notifier no-ops)    | eulogy/deaths channel                    |
| `OPENROUTER_API_KEY`        | (unset → always fallback)    | OpenRouter auth                          |
| `OPENROUTER_MODEL`          | `anthropic/claude-haiku-4.5` | generation model                         |
| `OPENROUTER_BASE_URL`       | `https://openrouter.ai/api/v1` | API base                               |
| `OPENROUTER_TIMEOUT_SECONDS`| `20`                         | per-call timeout → fallback on exceed    |

`.env` is git-ignored; `OPENROUTER_API_KEY` is provisioned by the operator. Until set, every post
uses the canned fallback automatically (no errors, channels never go silent).

## Testing approach (TDD)

Per repo convention: business rules in pure/Feature-tested services; Service/notifier wrappers stay
thin and uncovered (no gateway).

- **Unit (pure):** `LifeFactsBuilder` (ages, associates, linked); prompt building; placeholder →
  mention substitution (linked → `<@id>`, unlinked → backtick, missing-placeholder defense);
  headline/body split; fallback selection.
- **`Http::fake()`:** `OpenRouterClient` (success, non-2xx, timeout); `AnnouncementGenerator`
  (success path; timeout → fallback; empty/garbage → fallback; no-API-key → fallback).
- **Feature (`RefreshDatabase` + `CarbonImmutable::setTestNow()`):**
  - Ban gate: death at 59 min → no ban; at 60 min → ban; reroll suicides → no ban.
  - Grace gate: reroll chain produces exactly one birth + one eulogy (the worked example).
  - `go_live_at` + freshness gating for births and eulogies.
  - Idempotency: `birth_announced_at` / `eulogy_posted` prevent duplicate posts across ticks.
  - `AdmIngestor` writes `death_log` on a death; the excerpt contains the death window lines.
- **Config defaults** asserted in tests are pinned in `phpunit.xml` so live `.env` tuning can't turn
  the suite red.

## Out of scope / non-goals (YAGNI)

- Real character/skin/gender data (not in the logs) — gimmick dropped.
- Cross-ADM-file raw-log stitching for the death window.
- Bunker-exit dwell time, kill-streak callouts in eulogies beyond what facts already provide.
- Editing/refreshing a posted birth or eulogy (one-shot posts, unlike the leaderboard/roster).
- Queue/async infrastructure — generation runs inline in the 60s tick with a timeout + fallback;
  volume is low enough that this is simpler and sufficient.

## Known trade-offs (operator-visible)

- **Births are delayed ~5 minutes** by design — the cost of de-duping rerolls without an oracle. A
  genuine fresh spawn is announced once it survives the grace window.
- **OpenRouter is an external dependency** with cost + latency; the fallback path keeps the bot fully
  functional (just less custom copy) when it's unavailable or unconfigured.
