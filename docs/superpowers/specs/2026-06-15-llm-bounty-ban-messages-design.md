# LLM-Generated Bounty & Ban Channel Messages — Design

**Date:** 2026-06-15
**Status:** Approved (pending spec review)

## Summary

Extend OpenRouter LLM generation — already used for births/eulogies — to the **bounty**
and **ban** public channel announcements. Each affected channel post is generated by an
LLM, with the existing `MessagePicker` personality pool as a byte-for-byte fallback (no
`OPENROUTER_API_KEY` ⇒ canned copy, exactly like the lifecycle feature).

Two scope additions agreed during brainstorming:

1. **Ban-created messages** (death / manual / extended) and **ban-expired** (unban)
   messages are LLM-generated, not just the bounty messages.
2. **Death-bans now post to the bans channel.** Today the death-ban channel post is
   suppressed (`DiscordBanNotifier::postsToChannel()` returns `false` for `ban.death`)
   because the eulogy owns the public death announcement. We un-suppress it so a death
   produces **two** posts: the eulogy (story) in `EULOGY_CHANNEL_ID` and a separate ban
   notice (consequence) in `BANS_CHANNEL_ID`.

### Out of scope (decided)

- **DMs stay canned.** Only public channel posts are LLM-generated. DMs keep their
  existing `*.dm.*` pools and plain-gamertag text (one LLM call per event, lower cost).
- **No new channel env key.** Reuse the existing `BANS_CHANNEL_ID` (already carries
  manual/extended/unbanned posts; eulogies use the separate `EULOGY_CHANNEL_ID`).
- **No new enable flag.** Gating is purely on `OPENROUTER_API_KEY` presence, mirroring
  `AnnouncementGenerator`. (A `LLM_FLAVOR_ENABLED` kill-switch was considered and dropped
  as YAGNI — unset the key to disable.)

## Affected messages

All are **public channel posts**; all currently flow through `MessagePicker::pick(key, …)`.

| Domain | Key | Channel | Placeholder tokens |
|---|---|---|---|
| Bounty | `bounty.placed` | `BOUNTY_CHANNEL_ID` (→ `BANS_CHANNEL_ID` fallback) | `{{TARGET}}` |
| Bounty | `bounty.moved` | " | `{{TARGET}}` |
| Bounty | `bounty.claimed` | " | `{{KILLER}}`, `{{TARGET}}`, `{{TOKENS}}` |
| Bounty | `bounty.ended` | " | `{{TARGET}}` (**must stay neutral — no payout hints**) |
| Ban | `ban.death` | `BANS_CHANNEL_ID` | `{{WHO}}`, `{{REASON}}`, `{{EXPIRES}}` |
| Ban | `ban.manual` | `BANS_CHANNEL_ID` | `{{WHO}}`, `{{REASON}}`, `{{EXPIRES}}` |
| Ban | `ban.extended` | `BANS_CHANNEL_ID` | `{{WHO}}`, `{{REASON}}`, `{{EXPIRES}}` |
| Ban | `ban.unbanned` | `BANS_CHANNEL_ID` | `{{WHO}}`, `{{REASON}}` |

DM keys (`bounty.dm.*`, `ban.dm.*`) are unchanged.

## Architecture

### New: `app/Services/Llm/FlavorGenerator.php`

A single shared generator (Approach A from brainstorming) — both domains produce the same
shape (one short cheeky line + placeholders + canned fallback), so one class avoids
duplicating the OpenRouter/fallback/substitution plumbing.

Constructed like `AnnouncementGenerator`:

```php
public function __construct(private OpenRouterClient $client, private MessagePicker $picker) {}
public static function fromConfig(): self  // OpenRouterClient::fromConfig() + new MessagePicker()
```

One public method:

```php
/**
 * @param string $key       existing personality dot-key (e.g. 'bounty.placed', 'ban.death')
 * @param array  $facts     plain prompt data (gamertags, token count, reason) — NEVER coordinates
 * @param array  $tokens    placeholder => final channel value, e.g. ['target' => '<@123>', 'tokens' => 5]
 * @param string $fallback  the plain literal already passed to pick() today
 */
public function line(string $key, array $facts, array $tokens, string $fallback): string
```

**LLM path:**
1. Build a shared system prompt (voice = the server's announcer: one short, cheeky line;
   include the given placeholders verbatim; for `bounty.ended`, stay neutral and never
   hint at a payout / reward).
2. Build a per-key user instruction from `$key` + `$facts`.
3. `client->complete($system, $user)` → a single line containing `{{…}}` placeholders.
4. `strtr` placeholders → the `$tokens` channel values (`{{TARGET}}` → `<@123>`,
   `{{TOKENS}}` → `5`, `{{EXPIRES}}` → `<t:…:f>`, etc.). Trim and return.

**Fallback path:** on **any** `Throwable` (missing key, timeout, non-2xx, empty content,
unsubstituted-placeholder sanity failure), return
`picker->pick($key, $colonTokens, $fallback)` where `$colonTokens` is the `:token` form
derived from `$tokens` (`['target' => …]` → `[':target' => …]`). This is byte-for-byte
today's behavior.

**Why placeholders, not raw mentions in the prompt:** matches the lifecycle reasoning —
the LLM can mangle a raw `<@id>` / `<t:…>` string, so the model emits `{{…}}` tokens we
substitute deterministically afterward. The same generated line is *not* reused for the
DM (DMs stay canned), so no second-person variant is needed.

### Changed: `DiscordBountyNotifier` and `DiscordBanNotifier`

Each gains an optional constructor param:

```php
public function __construct(/* existing */, ?FlavorGenerator $flavor = null)
{
    // …
    $this->flavor = $flavor ?? FlavorGenerator::fromConfig();
}
```

(Default must be lazy — `fromConfig()` can't be a default param value.)

Every **channel** post changes from:

```php
$this->toChannel($this->picker->pick($key, $colonTokens, $fallback));
```
to:
```php
$this->toChannel($this->flavor->line($key, $facts, $tokens, $fallback));
```

DM sends (`toUser(...)` via `picker->pick('*.dm.*', …)`) are untouched.

**`DiscordBanNotifier` specifics:**
- Remove the `postsToChannel()` death-suppression guard (and its usage) so `ban.death`
  posts to the channel like the others. Update the stale "do NOT re-wire" comment on the
  `ban.death` pool in `config/personality.php`.
- The death-ban line stays consequence-framed (e.g. "benched until {{EXPIRES}}"), not a
  death-story retelling, so it doesn't duplicate the eulogy.

### Call sites — no changes required

The default constructor param means the 6 instantiation sites keep working unchanged:
`IngestAdmService`, `BanExpiryService`, `AdminBanCommand`, `UnbanCommand`,
`AdminUnbanCommand` (ban) and `BountyTickService` (bounty). In production
`FlavorGenerator::fromConfig()` builds the LLM-backed generator; with no API key it falls
straight through to the canned pools.

## Data flow (example: bounty placed)

```
BountyService::place()
  → DiscordBountyNotifier::placed(Bounty, Player $target)
      $mention = PlayerMention->forPlayer($target)         // channel mention
      $line = flavor->line(
          'bounty.placed',
          facts:    ['target_tag' => $target->gamertag],
          tokens:   ['target' => $mention],
          fallback: '🎯 A bounty is now on :target …'       // existing literal
      )
      → toChannel($line)
      → toUser(...) unchanged (canned bounty.dm.placed)
```

## Error handling

- `FlavorGenerator::line()` never throws — all LLM failures resolve to the canned pool.
- Notifiers remain best-effort: Discord send failures continue to no-op silently.
- A returned line still containing an un-substituted `{{…}}` placeholder is treated as an
  LLM failure → fall back to canned (guards against the model dropping/renaming a token).

## Testing (TDD — failing test first)

**`FlavorGenerator` (Feature/Unit, `Http::fake`):**
1. LLM returns a line with placeholders → assert final line has tokens substituted (e.g.
   the mention `<@123>` present, no residual `{{TARGET}}`).
2. LLM failure (no API key / HTTP 500) → deterministic fall-through to the canned pool
   line (inject a fixed `MessagePicker` chooser).
3. Un-substituted placeholder in LLM output → treated as failure → canned fallback.
4. `bounty.ended`: the prompt sent to OpenRouter (captured via `Http::fake`) contains the
   neutrality instruction, **and** the canned fallback stays neutral (preserves the
   existing associate-farm guard / `bounty.ended` neutrality test).

**Notifier tests:** run with no `OPENROUTER_API_KEY` (suite default), so channel posts hit
the canned path and existing assertions continue to hold. Update the single
death-suppression test to assert death-bans now post to the channel (and that the eulogy
path is unaffected).

**Config defaults** pinned in `phpunit.xml` already cover the `OPENROUTER_*` block; no new
tunable to pin.

## Conventions honored

- Logic in a testable plain service (`FlavorGenerator`); notifiers stay thin wrappers.
- Public channel posts may mention; DMs keep plain gamertag text (split preserved).
- LLM is best-effort and never throws into callers; canned copy is the safety net.
- Not gated by `BAN_DRY_RUN` (channel posts are independent of real Nitrado bans) — note
  death-ban *channel posts* now fire whenever a death-ban is recorded, same gating as the
  ban record itself.
